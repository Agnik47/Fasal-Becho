# DATABASE.md

# Fasal Becho — Database Design

Version: 1.0

Status: MVP Database Specification

This document defines the logical data model for Fasal Becho.

It is **not** a Prisma schema or SQL definition.

Instead, it documents:

- Business entities
- Entity responsibilities
- Relationships
- Validation rules
- Data ownership
- Lifecycle
- Future expansion

This document is the single source of truth for the application's data model.

---

# Database Philosophy

The database should represent **business facts**, not business logic.

Store only information that is necessary.

Business rules belong in the Business Layer, not inside the database.

The database should be:

- Simple
- Normalized
- Easy to extend
- Easy to query
- Easy to migrate

# Entity: Enterprise

## Purpose

Represents a rural micro-enterprise.

This is the central entity of the application.

Every prediction is generated for one enterprise.

---

## Responsibilities

- Business profile
- Sector information
- Location
- Financial ownership

---

## Core Fields

- ID
- Enterprise Name
- Owner Name
- Business Type
- District
- State
- Village
- Pin Code (required — the Weather Service, `ARCHITECTURE.md` Section 8, resolves this to lat/long for the Meteostat pull in Prediction Pipeline Step 4; see `Sources Reference.md` Section 2)
- Registration Number (optional)
- Created By
- Created At
- Updated At

---

## Relationships

One Enterprise

↓

Many Financial Records

↓

Many Predictions

↓

Many Recommendations

↓

Many Officer Actions

# Deletion Rules

Deleting an Enterprise should **not** immediately remove historical data.

Preferred approach:

Soft Delete

Historical predictions and officer actions should remain available for audit purposes.

---

# Validation Rules

The database should enforce:

- Required foreign keys
- Valid enum values
- Positive financial values
- Unique primary keys
- Referential integrity

Business validations belong in the Business Layer.

---

# Future Database Expansion

The following entities are intentionally excluded from the MVP:

- Notification
- District Analytics
- Commodity Cache
- Weather Cache
- Portfolio Dashboard
- Audit Logs
- Account Aggregator
- Bank Accounts
- Loan Applications
- Document Uploads
- Multi-sector Configuration

These may be introduced in future versions without changing the existing entity relationships.

---

# Database Design Principles

The database should always prioritize:

- Simplicity over complexity
- Readability over optimization
- Historical accuracy over overwriting data
- Clear ownership of entities
- Predictable relationships
- Easy extensibility

Every table should have a single, well-defined responsibility.

---

# Guiding Principle

The database exists to store trustworthy business facts.

It should never contain business rules, prediction logic, or AI decision-making.

Those responsibilities belong to the Business Layer defined in ARCHITECTURE.md.
